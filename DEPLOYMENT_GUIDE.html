INNEED Production Deployment

Vercel + Railway + Neon + Upstash — ~$5/month

Frontend: Vercel (free) Backend: Railway (~$5/mo) DB: Neon (free tier)
CTO Verdict: This is the right stack for MVP. Vercel was built by the Next.js team — it's the best place for your frontend. Railway runs your Docker backend without ops overhead. Neon gives you real PostgreSQL for free. Total fixed cost: ~$5/month (just Railway compute).
2 Code Fixes Already Applied: frontend/next.config.ts now proxies /api/* → Railway backend (fixes cross-origin cookie auth). backend/prisma/schema.prisma now has directUrl for Neon migration compatibility.

Architecture

Browser │ ▼ inneed.in (Vercel CDN — global edge network) │ ├─ / → Next.js SSR/CSR pages │ └─ /api/* → Vercel Edge Rewrite (built-in, free) │ strips /api, forwards to Railway ▼ inneed-api.up.railway.app (Fastify, Docker) │ ┌──────┴──────────────┐ │ │ Neon PostgreSQL Upstash Redis (ap-south-1) (ap-south-1) Pooled conn Rate limiting + Direct conn External services (outbound from Railway only): → Razorpay payments.razorpay.com → Cloudinary api.cloudinary.com → MSG91 api.msg91.com → Resend api.resend.com → Sentry o*.ingest.sentry.io → Google OAuth oauth2.googleapis.com

Why rewrites fix cookies

Your backend sets cookies with sameSite: 'strict'. Without the rewrite, the browser talks to Vercel (inneed.in) AND Railway (different domain) — strict cookies won't be sent cross-origin and auth breaks silently. With the rewrite, the browser only ever talks to inneed.in. Vercel's edge proxies the request to Railway server-side. The Set-Cookie arrives on inneed.in. Auth works perfectly with zero backend changes.

Code Changes Already Applied

Two files have been patched. Here's what changed and why.

1. frontend/next.config.ts — API proxy rewrite

Added inside nextConfig. Vercel's edge rewrites this server-side — no browser ever talks directly to Railway:

async rewrites() {
  return [
    {
      source: '/api/:path*',
      destination: `${process.env.BACKEND_URL}/:path*`,
    },
  ]
},

BACKEND_URL is a server-side env var (no NEXT_PUBLIC_ prefix) — set it in Vercel dashboard to your Railway URL. It is never exposed to the browser.

2. backend/prisma/schema.prisma — Neon directUrl

Neon's pooled connections cannot run schema migrations. Prisma needs a direct (non-pooled) connection for prisma migrate deploy:

datasource db {
  provider  = "postgresql"
  url       = env("DATABASE_URL")    // pooled connection — runtime queries
  directUrl = env("DIRECT_URL")      // direct connection — migrations only
}

Vercel Environment Variables

Set these in Vercel → project → Settings → Environment Variables. Apply to Production, Preview, and Development as appropriate.

NEXT_PUBLIC_* vars are baked into the browser bundle at build time. Changing them requires a redeploy. Non-prefixed vars (BACKEND_URL) are server-side only and never sent to the browser. 
# ── Server-side only (never exposed to browser) ──────────────────────────────────
BACKEND_URL=https://inneed-production-xxxx.up.railway.app
# The Railway backend URL (no /api suffix — rewrites add the path)

# ── Client-side (baked into bundle at build time) ────────────────────────────────
NEXT_PUBLIC_API_URL=/api
# Relative path — axios sends /api/auth/login → Vercel rewrite → Railway /auth/login

NEXT_PUBLIC_APP_URL=https://inneed.in
# Used for SEO meta tags, Open Graph URLs, auth redirects

NEXT_PUBLIC_GOOGLE_CLIENT_ID=XXXXXXXXXX.apps.googleusercontent.com
# Same as backend GOOGLE_CLIENT_ID — must match

NEXT_PUBLIC_GOOGLE_MAPS_API_KEY=AIzaXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX
# Google Maps Platform → APIs & Services → Credentials → API key
# Restrict key: HTTP referrers → https://inneed.in/*

NEXT_PUBLIC_RAZORPAY_KEY_ID=rzp_live_XXXXXXXXXXXXXXXX
# Public key only — safe to expose. Backend uses the secret key.

NEXT_PUBLIC_SENTRY_DSN=https://XXXXXXXX@XXXXXXXX.ingest.sentry.io/XXXXXXXX
# Can be same DSN as backend, or a separate frontend project in Sentry

# ── Sentry source map upload (optional but recommended) ──────────────────────────
SENTRY_AUTH_TOKEN=sntrys_XXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX
# Get from: sentry.io → Settings → Auth Tokens → Create new token

Database Migrations

Automatic (on every deploy)

The Railway start command runs migrations before starting the server:

cd /app/backend && npx prisma migrate deploy && node dist/index.js

prisma migrate deploy is safe to run on every deploy — it checks which migrations have already been applied and skips them. If there are no new migrations, it starts immediately.

Manual migration (Railway Shell)

# Run from Railway service Shell tab
cd /app/backend

# Apply pending migrations
npx prisma migrate deploy

# Check migration status
npx prisma migrate status

# One-time seed (run only once after first deploy)
npx tsx prisma/seed.ts

Safe migration rules

RuleWhy
Only migrate deploy in production, never migrate devmigrate dev can drop and recreate tables
Write additive migrations (add columns, never drop in same PR)Old app containers may still be running during rollout window
New NOT NULL columns must have a DEFAULT valueExisting rows need a value; old app code doesn't send it
Test migrations on Neon staging branch firstCatch breakage before it hits production

Automated Deploys

Both platforms auto-deploy on every push to main. No CI/CD configuration needed beyond connecting your GitHub repo.

EventVercelRailway
Push to mainRebuilds Next.js, deploys to CDN (~2 min)Rebuilds Docker image, replaces container (~4 min)
PR openedCreates preview URL (pr-42-inneed.vercel.app)No preview by default
PR merged to mainProduction deploy triggered automaticallyProduction deploy triggered automatically
GitHub Actions CI still runs on every push (lint + build check). If it fails, Railway/Vercel will still deploy (they don't depend on Actions). To block bad deploys, add a Railway deploy hook that waits for CI — but this is optional for early stage.

Rollback

Frontend (Vercel) — one click

Vercel stores every deployment. Go to Vercel dashboard → project → Deployments → find the last good deploy → click ⋯ → Promote to Production. Done in 30 seconds.

Backend (Railway) — redeploy previous SHA

Railway dashboard → service → Deployments tab → find the last good deploy → click Redeploy. Railway will rebuild from that git SHA.

Code rollback (via git)

# Revert the bad commit and push
git revert HEAD --no-edit
git push origin main
# Both Railway and Vercel auto-deploy from the revert commit

Database rollback

There is no automatic database rollback. Prisma does not support down migrations. Strategy:

  • If migration added columns → safe to rollback app code (old code ignores new columns)
  • If migration dropped columns → write a new forward migration that re-adds them, push to main
  • For catastrophic data loss → restore from Neon's point-in-time recovery (available on paid plans)

Post-Deploy Smoke Test Checklist

Run through this after every production deploy:

Backend

  • curl https://inneed.in/api/health returns {"status":"ok"}
  • curl https://inneed.in/api/items?limit=3 returns listing array
  • Sentry dashboard shows no new errors
  • UptimeRobot monitor shows UP
  • Railway metrics show container running (not restarting)

Frontend

  • Homepage loads at https://inneed.in
  • Hero search works → navigates to /search with results
  • Item detail page loads images from Cloudinary
  • Login page renders correctly
  • Google OAuth button appears (checks NEXT_PUBLIC_GOOGLE_CLIENT_ID)
  • No console errors in browser DevTools
  • Network tab: /api/* requests return 200 (not 502 or CORS errors)

Auth flow

  • Register new user → succeeds, redirects to home
  • Login with test account (from seed) → succeeds, user dropdown appears
  • Refresh page → user stays logged in (cookie persists)
  • Logout → user cleared, auth-only pages redirect to login

Vendor flow

  • Login as vendor account (from seed) → vendor dashboard accessible
  • Listings page loads vendor's listings

Admin flow

  • Login as admin account (from seed) → admin area accessible
  • Vendor approval page loads
